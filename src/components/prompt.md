Overview
A Claude Code slash command /plan that runs a local interview via Ollama, assembles enriched context, fires a single Opus API call, and writes a PLAN.md the executing model can follow without ambiguity.

File structure
.claude/
commands/
plan.md          ← slash command definition + prompt logic
scripts/
plan-runner.js   ← Node script: interview → context → API call → write output
prompts/
interviewer.md   ← system prompt for local interview phase
planner.md       ← system prompt sent to Opus
.claude.md           ← project context (already exists, read during assembly)
PLAN.md              ← output, written after Opus call
The command definition (plan.md) is lightweight — mostly it invokes plan-runner.js with the task string and mode flag. All real logic lives in the Node script so it's testable independently of Claude Code.

Entry point — the slash command
Invocation signatures:
/plan "refactor auth module"           ← small, one-shot (skips interview)
/plan --think "redesign auth module"   ← large, runs interview first
The --think flag is the only branching decision the user makes upfront. Everything else is handled internally.
The command passes two things to plan-runner.js: the raw task string and the mode flag. That's it.

Mode A — small task (one-shot)
No interview. The runner immediately assembles context from:

The task string
CLAUDE.md contents
Current working directory file tree (1 level deep)
Any files explicitly mentioned in the task string (auto-detected)

Then fires Opus. Total user interaction: one line.

Mode B — large task (interview first)
This is the key design. The runner spins up a short Ollama call with the interviewer system prompt before touching the Opus API.
The interviewer system prompt (interviewer.md)
The system prompt instructs Ollama to produce exactly 3–5 clarifying questions, never more. The questions must meet a filter: would the answer materially change the plan? If not, don't ask it.
Question categories to target:

Constraints ("what must this not break?")
Existing patterns ("is there a pattern in the codebase to follow, or greenfield?")
Scope boundaries ("what is explicitly out of scope?")
Success criteria ("how will you know this plan worked?")
Dependencies ("anything this touches that lives outside this repo?")

Questions to explicitly avoid:

Language / framework (already in context)
Things answerable by reading the file tree
Generic "what's your timeline" filler

The interviewer returns questions as a numbered list. The runner prints them, collects answers via stdin (simple line-by-line), and stores them as a structured Q&A block.

Context assembly
Before calling Opus, the runner builds a single context document in this order:
[task]
Raw task description as given by the user.

[project context]
Contents of CLAUDE.md, trimmed to relevant sections if long.

[file tree]
Top-level structure, 2 levels deep. Filtered to exclude node_modules, .git, dist.

[relevant files]
Any files auto-detected from task string or explicitly flagged.

[interview qa]           ← only in --think mode
Q: <question>
A: <user answer>
... for each Q&A pair

[constraints]
Distilled hard constraints extracted from interview answers. Runner does a
simple pass to pull explicit "must not", "should not", "has to" statements.
This document is what gets sent to Opus as the user message. The goal is that Opus never has to guess — everything it needs is in the context block.

The Opus call
Single API call to claude-opus-4-6 (or latest Opus). Max tokens set generously — plans can be long.
The planner system prompt (planner.md)
This is the most important piece. Opus is instructed to return a plan in a strict structure:
markdown## assumptions
What I am assuming to be true that isn't stated. If any assumption is wrong,
the plan may need revision.

## decision log

Key architectural or approach decisions, with brief rationale. Not steps —
just the choices made and why.

## constraints

Hard constraints extracted from context. Things the execution must not violate.

## steps

Ordered implementation steps. Each step must be:

- Atomic (one clear thing to do)
- Verifiable (you know when it's done)
- Referenced to a file or component where possible

## risks

Things likely to go wrong during execution, and what to do if they do.
The system prompt explicitly forbids: vague steps ("update the codebase"), undeclared assumptions, and steps with no concrete location. If Opus can't be specific, it must say so in the assumptions section rather than papering over it.

Output — PLAN.md
Written to project root after Opus responds. The runner prepends a header:
markdown<!-- generated by /plan — do not edit manually -->

<!-- task: <original task string> -->

<!-- model: claude-opus-4-6 -->

<!-- date: <iso timestamp> -->

This makes it easy to know when the plan is stale and what it was generated for.
The executing model (Ollama) is expected to read PLAN.md at the start of the session. This can be enforced by adding a reference to it in CLAUDE.md:
markdown## active plan
If PLAN.md exists in the project root, read it before beginning any work.
Treat it as authoritative for task scope and approach decisions.

Cost control
The runner logs token usage after the Opus call to stdout:
[plan] Opus call complete
input:  3,241 tokens  ($0.016)
output: 1,847 tokens  ($0.046)
total:  ~$0.062
Cheap enough to be guilt-free, visible enough to track over time. If you want to keep a running total, the runner can append to .claude/plan-usage.log.

What's not in scope (deliberately)

No interactive back-and-forth with Opus — that's what the interview phase is for
No automatic file reading beyond what's mentioned in the task — keeps context lean
No plan versioning — PLAN.md is always overwritten; if you want history, commit it to git
No auto-execution — the plan is advisory, not a CI trigger

Build order

planner.md system prompt — most important, get this right first
plan-runner.js skeleton — entry, mode branch, stdout/stdin wiring
Context assembler function — reads CLAUDE.md, file tree, detected files
Interviewer integration — Ollama call + Q&A collection
Opus API call + response parser
PLAN.md writer
plan.md slash command definition — thin wrapper calling the script
Test on a real task, tune the planner prompt based on output quality
